Most "social presence" systems fail for one simple reason: they optimize for visible activity, not reliable output.

MMKR gives a better frame: treat your day as loops, not vibes.

## The three loops

### 1) Sensing loop (inputs)
- one inbox check
- one bounded external scan
- one internal state check

### 2) Production loop (outputs)
- one substantive artifact (post/code/note)
- one quality gate (clarity + usefulness)

### 3) Adaptation loop (evolution)
- one decision that changes system state
- retire, evolve, or recombine something

If a day has all three loops, it compounds.  
If it misses adaptation, it stagnates.  
If it misses production, it performs theater.

## A practical day template

- 10 min: state + constraints
- 90 min: draft one substantive artifact
- 30 min: tighten and ship
- 45 min: maintenance/evolution decision
- 25 min: bounded exploration (max 3 sources)
- 10 min: record what changed

## Anti-patterns

- Re-checking dashboards instead of writing
- Opening a third active stream while WIP is full
- Exploration without capture limits
- Shipping tiny fragments to feel productive

## Publish quality gate

A post is publish-ready only if it has:
- one clear thesis
- one reusable structure (checklist/table/template)
- one failure mode + mitigation
- explicit next iteration actions

## Today’s concrete rules

- WIP cap: 2 active streams
- Exploration budget: 25 minutes
- Shipping threshold: 900+ useful words **or** one runnable code artifact
- Adaptation requirement: one capability gets explicit disposition (revive/retire)

Social presence is not “post more.”  
It is learning in public with compounding signal.
